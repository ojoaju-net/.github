# Knowledge Graph Versioning and Traceability

## 1. Versioning Philosophy

To preserve the sovereignty and mathematical integrity of a user's professional and academic record, **Ojoaju** implements a **linear, immutable, append-only version history**.

Instead of treating the Knowledge Graph (KG) as a mutable database that is overwritten with each change, the KG is treated as a chronological sequence of cryptographic state transitions (commits).

### Core Rules of Ojoaju Versioning:

1. **Linear History:** No branches, merges, or forks are permitted on the active profile node. The timeline is a simple linked list where each version $V_n$ points to its parent $V_{n-1}$.

2. **Immutability:** Once a version is committed to the cloud node with a valid cryptographic signature, it cannot be edited, deleted, or reordered.

3. **Roll-Forward Recovery:** Reverting to a past state is not done by deleting the present. Instead, a new forward commit is appended whose net effect yields the target historical state.

## 2. Commit Manifest Schema

Every state transition is wrapped in a **Commit Manifest**. This manifest contains the patch delta, metadata, and the cryptographic proof of authorization.

```
{
  "version_id": "sha256_hash_of_commit_body",
  "parent_id": "sha256_hash_of_parent_commit | null",
  "timestamp": "2026-06-20T19:30:00Z",
  "author_node_id": "did:ojoaju:6f8a9b123456789abcdef",
  "patch": [
    { "op": "replace", "path": "/knowledge_graph/projects/0/role", "value": "Principal Architect" }
  ],
  "signature": "ed25519_signature_hex"
}
```

### Hash Verification

The `version_id` is computed by hashing the canonicalized JSON representation of the commit body (excluding the `signature` field):

$$\text{version\_id} = \text{SHA256}(\text{CanonicalJSON}(\text{parent\_id} + \text{timestamp} + \text{author\_node\_id} + \text{patch}))$$

The `signature` is generated by the local client using the user's Secret Key ($SK$):

$$\text{signature} = \text{Ed25519\_Sign}(SK, \text{version\_id})$$

## 3. State Reconstruction Algorithm

To display the current or any past state of the profile, the node parses the chain of commits starting from the Genesis Commit ($V_0$, where the graph is empty `{}`) up to the target version $V_{\text{target}}$.

$$\text{State}(V_{\text{target}}) = \text{ApplyPatches}(\{\}, V_0 \to V_1 \to \dots \to V_{\text{target}})$$

```
graph TD
    G[Genesis:  { } ] -->|Apply V1 Patch| V1[State V1]
    V1 -->|Apply V2 Patch| V2[State V2 (Current)]
```

By storing only the patches (deltas) instead of complete graph snapshots, storage on the cloud node is heavily optimized, and changes are easily auditable.

## 4. Roll-Forward Recovery (Reverting Changes)

If a user wishes to restore an old profile state (for example, recovering data deleted in a past update), the protocol forbids rewriting history.

To revert the current state $V_{\text{current}}$ back to a target historical state $V_{\text{target}}$:

1. The local client reconstructs the JSON state of both $V_{\text{current}}$ and $V_{\text{target}}$.

2. It computes a **reverse diff** between the two states using the JSON Patch standard.

3. It packages this reverse diff into a new commit $V_{\text{current}+1}$.

4. When applied, $V_{\text{current}+1}$ brings the active graph back to the exact data structure of $V_{\text{target}}$, while keeping the chronological record intact.

$$\text{Patch}(V_{\text{current}+1}) = \text{ComputeJSONDiff}(\text{State}(V_{\text{current}}), \text{State}(V_{\text{target}}))$$

### Diagram of Roll-Forward to V1 from V2:

```
graph LR
    V1[State V1] -->|Applied V2 Delta| V2[State V2]
    V2 -->|Applied Reverse Patch V2 to V1| V3[State V3 (Clone of V1)]
```

This mechanism ensures that the cryptographic chain of custody remains unbroken and that vector database indexes can be incrementally updated without full rebuilds.
